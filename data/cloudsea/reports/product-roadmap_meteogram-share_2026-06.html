1. 竞品与数据源研判

1.1 截图产品形态识别

截图特征：顶栏「日出日落月相」、底栏 meteoblue 品牌、太子尖 30.18°N 118.90°E · 1557 m asl、约 7 日逐时、四行面板：
  • 温度 + 天气图标（日/夜背景色带）
  • 降水柱状 + 湿度折线 + 降水概率
  • 垂直云量剖面（高度 0–14 km 灰度云密，含站点海拔参考线）— 对云海最关键
  • 风速/阵风折线 + 风向箭头
该类界面在行业中通常称为 Meteogram（气象ogram）/ All-in-One，meteoblue 文档对应产品为 meteogram_one、crossSection_clouds 等 Image/Vis API。

1.2 厂商侧数据源（meteoblue 公开文档归纳）

层级内容说明
核心 NWP NEMS 族（NEMS4 中欧 4km、NEMS12、NEMSGLOBAL 30km 等） 自研/运营区域高分辨率 + 全球背景场；强调地形与云发展
地形 DEM ~80m + 请求参数 asl（海拔米） 同经纬度不同海拔预报分离（山谷 vs 山顶）
同化/后处理 地面站、雷达、卫星；历史预报渐进订正 近时效预报随观测融合，非单纯模式直出
交付形态 Forecast API（JSON）、Images API（meteogram PNG/SVG）、Maps Tile API App 可能嵌入图片 API 或自绘 + 自家后端聚合
商业 API Key 计费、域名白名单、调用量限制 需商务评估 不适合无预算直接替换 Open-Meteo

参考：meteoblue Datasets · Forecast Images / Meteogram

1.3 国内同类 App 常见组合（推断，需商务核实）

  • 白标 meteoblue / Windy / 彩云 图表 + 自有 POI/摄影内容
  • 纯 Open-Meteo / ECMWF 开放数据 自研 UI（成本低，垂直云图需自算）
  • 景区合作站 实况探头（稀缺，难覆盖全国山头）

2. 本项目现状与差距矩阵

能力维度yunhai 现状截图竞品差距
预报时长 5 天 × 24h（120 点） 约 7 天逐时 中 可扩至 7d（Open-Meteo 支持，需评估负载）
海拔订正 观景点 elevation + DEM 地形因子；Open-Meteo 网格点 2m 场 显式 asl=1557 模式层订正 中 未向模式传递「预报海拔」参数（Open-Meteo 无 asl，靠后处理）
垂直云结构 低/中/高云量 + 850/700/925hPa 湿度/逆温；无 0–14km 剖面图 云量垂直剖面（核心卖点） 大 展示与机理解释弱
风场展示 10m 风速因子；无风向箭头时间条 风速/阵风 + 风向箭头 中 数据已有 wind，UI 未展开展示
云海/日出预测 规则 fuzzy v2 + 点位 ML + 地形可观域；AI 解读 通用气象，未必有云海概率 优势 垂直领域评分 + 标注闭环
卫星 Himawari 红外裁切 + 当天轻校正 meteoblue 卫星图层（全球产品） 部分优势 东亚日出场景已接入
可解释/分享 因子列表、雷达图、AI 文字；无 对外分享页 截图分享为主 大 缺「带依据的分享」
结论：竞品强在通用气象可视化（尤其垂直云剖面 + meteogram）； 我们强在云海/日出领域评分、ML 标注、地形观云模式。 不宜整体替换数据源；宜 Open-Meteo 深化 + 自研云海图层 + 可选商业补强。

3. 可借鉴方案（提高精度 + 展示效果）

方案 A · 零增量成本（推荐 P0 部分）

数据源可达
  • 垂直云剖面（自绘）：扩展 Open-Meteo 请求 cloud_cover_*hPa + geopotential_height_*hPa（多气压层）， 将云量映射到高度轴；叠加观景点海拔红线、估算云底线。
  • Meteogram 面板：前端 ECharts 四行（温/降水/剖面/风），复用现有 120 点 JSON。
  • 风向：补充 wind_direction_10m（Open-Meteo 已有变量，当前未请求）。
  • 日峰值云海：日摘要改为「日出窗口峰值」避免与卡片 27% 矛盾（已发现过的产品 bug 类问题）。

方案 B · 中等成本（P1）

需开发 + 缓存
  • 模式选择：Open-Meteo models=icon_seamless 或 regional 对比 ECMWF，A/B 显示「模式来源」。
  • ICON 压力层云图：DWD ICON API 与现有 850/700 场交叉验证。
  • 剖面 → 评分：云底低于观景点且谷地湿度高 → 上调 plausibility；反之压制（与 terrain 模块联动）。
  • 分享卡片：见第 5 节。

方案 C · 商业气象 API（P2 可选）

付费 / 合规
  • meteoblue Forecast + crossSection_clouds：快速达到截图级剖面（嵌入图或 JSON）。
  • Windy Point Forecast API：类似 alt 订正（需授权）。
  • 建议：仅对精选景区试点，控制调用量；保留 Open-Meteo 为默认免费链路。

方案 D · 观测增强（P2+）

难规模化
  • 景区微气象站、摄像头云况（非公开数据，合作成本高）。
  • Himawari 云顶高度反演（科研级，工程量大）。

4. 数据源可达性、成本与合规

数据源可达性关键变量限制本项目
Open-Meteo Forecast 已用 2m 场、分层云、能见度、850/700/925、可扩展 wind_direction、多层 cloud_cover 非商业无限；需遵守 fair use；无 asl 参数 主链路，免费
Open-Meteo Historical Forecast 已用 标注回填、LOOCV 2021+；与训练一致性问题已文档化 ML 闭环
Copernicus DEM / Elevation API 已用 海拔、地形快照 — terrain_store
Himawari GIBS 已用 红外云图、区域云量 无未来；仅过去/临近实况 地图蒙版
meteoblue API 需签约 Meteogram 图、NEMS 预报、垂直云剖面 按量计费；中国区网络需测 P2 可选
天地图 已用 底图 POI 浏览器 Key；不涉及预报 前端
DeepSeek LLM 已用 出行解读 Token 成本；须与数值结论一致 辅助，非预报源

5. 「分享预测 + 依据」功能设计

产品目标：用户一键生成可转发链接/图片，包含 结论（云海/日出/综合） + 可追溯依据（非黑盒）， 提升传播与信任；默认免责声明与「非商业预报承诺」。

5.1 用户故事

  • 摄影师将「6 月 6 日 五女山点将台 · 谨慎前往 · 云海 27%」分享到微信群，好友点开可见依据摘要。
  • 依据包含：日出窗口逐时表、规则%/ML%（若启用）、关键因子 Top3、AI 解读节选（可选）、卫星云图缩略（若有）。

5.2 功能点分解

ID功能说明优先级
F1生成分享按钮「分享本日预测」；携带 spot/vp/date + 预测快照 hashP0
F2分享页 /s/{id}SSR 或静态 JSON + 轻量 Vue；SEO/OG 标签P0
F3依据面板结构化 Evidence，非整页 PDF；可折叠「气象/规则/ML/地形」P0

5.4 与 AI 解读的关系

  • 分享依据以结构化 Evidence JSON 为准（与 llm_advisory 的 context 同源）。
  • AI 文案作为可选附录，须标注「AI 辅助」；禁止分享页仅一张 AI 截图无数字依据。
  • 复用现有 prediction_pipeline、verdict_hint，保证与主页一致。

8. 风险、依赖与决策门禁

风险影响缓解
Open-Meteo 压力层云量由 RH 推算 垂直剖面与 meteoblue 视觉不一致 标注「模式估算剖面」；不声称雷达级精度
分享快照与实时预报不一致 用户质疑「链接与 App 不同」 冻结时间戳 + 明确「分享时预报」
无 ML 点位误展示 ML 信任受损 ml_active 门禁（AI/分享共用）
meteoblue 成本 预算超支 Phase 3 可选；默认不自研依赖
多气压层 API 体积增大 预测变慢 剖面数据独立缓存键；与 predict 并行或懒加载
请您确认（回复时可勾选）：
  1. 是否同意 Phase 1a（自研 Meteogram + Open-Meteo 多层云） 优先于采购 meteoblue？
  2. 是否同意 Phase 1b（分享 + 依据 MVP） 与 1a 并行或紧随其后？
  3. 预报天数是否从 5 天扩展到 7 天？（影响缓存与前端时间轴）
  4. 分享链接是否需要登录/审核，还是完全公开只读？

确认后下一步：输出 Phase 1 技术任务拆解（Issue 级）并开始实现，仍可按模块分批上线。
